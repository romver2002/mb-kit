# Ранбуки {{PROJECT_NAME}}

Пошаговые инструкции «что делать, когда сломалось». Пишутся так, чтобы дежурный
(или агент) в 3 часа ночи выполнил их без размышлений: конкретные команды,
конкретные ссылки, конкретные пороги.

## Правило

**После каждого инцидента — ранбук.** Инцидент закрыт только тогда, когда:

- по новому типу отказа создан ранбук из шаблона
  [incident-template.md](incident-template.md), **или**
- существующий ранбук обновлён тем, что узнали (шаг не сработал, команда
  устарела, нашлась более быстрая диагностика).

Ранбук создаётся в течение 1–2 дней после инцидента отдельным PR —
пост-фактум-шаг процесса срочного исправления
[../../workflows/hotfix.md](../../workflows/hotfix.md);
куда смотреть во время инцидента — [../observability.md](../observability.md).

## Соглашения

- Один ранбук = один класс симптомов (не «всё про БД», а «БД: закончились соединения»).
- Имя файла: `NNN-краткий-симптом.md` (kebab-case, по симптому, а не по причине —
  ищут по тому, что видят).
- Ранбук устарел (инфраструктура изменилась) → обновить или пометить
  `status: archived` по правилам [../../_meta/lifecycle.md](../../_meta/lifecycle.md).
- Агент может создавать и обновлять ранбуки автономно в PR; проверка шагов
  на живом окружении — за человеком (ограничения — [../environments.md](../environments.md)).

## Реестр

Добавил ранбук → добавь строку здесь (в том же PR).

| Ранбук | Симптом | Серьёзность | Обновлён |
|---|---|---|---|
| Пример (замените своим): `001-api-5xx-spike.md` | всплеск 5xx на API, алерт `HighErrorRate` | критично | 2026-07-05 |
| TODO(template): следующий ранбук | | | |

TODO(template): строка выше — иллюстрация формата; файл `001-api-5xx-spike.md`
не существует, замените первым реальным ранбуком и сделайте имя ссылкой.

## Вопросы заполняющему

- Вспомни последние 2–3 инцидента или «страшных» места системы — для каждого
  уже можно завести ранбук, не дожидаясь повторения.
- Какой отказ наиболее вероятен на этой неделе? Есть ли на него ранбук?
