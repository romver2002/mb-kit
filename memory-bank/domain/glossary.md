# Глоссарий {{PROJECT_NAME}}

Канонический словарь предметной области (ubiquitous language). Здесь фиксируется,
как термин звучит в разговоре с бизнесом и как он **обязан** называться в коде.

## Правило единого языка

- **Термин в коде обязан совпадать с глоссарием.** Класс, таблица, поле API,
  событие — везде используется `Имя в коде` из таблицы ниже, без синонимов.
- Нашёл в коде синоним или «почти тот же» термин — это противоречие; действуй
  по [../_meta/lifecycle.md](../_meta/lifecycle.md) (раздел «Противоречия»).
- Переименование термина = правка этой таблицы + переименование в коде
  в одном PR (владелец раздела — см. [../_meta/governance.md](../_meta/governance.md)).
- Один термин — одна строка. Если значение зависит от контекста, заведи две строки
  с разными значениями колонки «Контекст» (сами контексты —
  в [bounded-contexts.md](bounded-contexts.md)).

## Как пополнять глоссарий

Чек-лист при добавлении термина:

- [ ] Термин звучит так, как его произносит бизнес/заказчик (не выдуман разработчиком)?
- [ ] Значение сформулировано в одном-двух предложениях, без отсылок «см. код»?
- [ ] `Имя в коде` — английское, в нотации из
      [../architecture/code-style.md](../architecture/code-style.md)?
- [ ] Указан контекст из [bounded-contexts.md](bounded-contexts.md)?
- [ ] Нет ли уже строки-синонима? (Синонимы запрещены — выбери один термин,
      остальные упомяни в значении: «также называют …, в коде не используется».)

Наводящие вопросы для стартового наполнения:

- Какие 10–15 слов чаще всего звучат на встречах с заказчиком?
- Какие сущности есть в схеме БД / основных модулях? Совпадают ли их имена
  со словами бизнеса? Расхождения — кандидаты в первую очередь.
- Есть ли слова, которые бизнес и разработка понимают по-разному? Зафиксируй
  оба значения по контекстам.
