# ADR-0001: Переход на Clean Architecture + SOLID

> Пример ADR — при адаптации шаблона замените деталями вашего реального решения
> или оставьте, если мотивировка совпадает.

## Контекст

{{PROJECT_NAME}} вырос из прототипа: бизнес-логика перемешана с доступом к данным
и транспортом, зависимости между модулями хаотичны. Действующие силы:

- **Рост связанности**: изменение в одном модуле ломает соседние; оценки сроков
  непредсказуемы, регрессии всплывают в неожиданных местах.
- **Сложность поддержки**: новый разработчик не может сказать, где живёт
  бизнес-правило — в контроллере, сервисе или запросе к БД.
- **AI-driven разработка**: значительную часть кода пишут агенты (Claude Code,
  Codex). Агенту нужна предсказуемая структура: одинаковые правила размещения
  кода во всех проектах группы, проверяемые границы, локальный контекст задачи.
- Тесты бизнес-логики требуют поднятой БД и внешних сервисов — медленно и хрупко.

## Рассмотренные варианты

| Вариант | Плюсы | Минусы |
|---|---|---|
| 1. Оставить как есть | Ноль затрат сейчас; команда привыкла | Связанность продолжает расти; каждая фича дороже предыдущей; агенты воспроизводят хаос и усугубляют его |
| 2. Слоистая архитектура (UI → Service → DAO) | Дешевле внедрить; знакома всем | Зависимости направлены вниз к БД: домен зависит от инфраструктуры; логика «стекает» в сервисы-простыни; границы не проверяемы |
| 3. Clean Architecture + SOLID | Домен изолирован и тестируем без инфраструктуры; направление зависимостей — единое проверяемое правило; одинаковая структура во всех проектах группы — предсказуемость для людей и агентов | Больше кода-обвязки (интерфейсы, DTO, мапперы); порог входа; дисциплина на ревью |

## Решение

Принимаем **Clean Architecture с явными границами слоёв** и SOLID как базовые
принципы дизайна для всех проектов группы на {{STACK}}.

- Четыре слоя: domain, application, infrastructure, presentation; внешние
  (infrastructure, presentation) зависят от внутренних (application, domain),
  зависимости направлены только внутрь, к домену.
  Детали и ответственность слоёв — [../architecture/overview.md](../architecture/overview.md).
- Домен не импортирует ничего из внешних слоёв и фреймворков; выход наружу —
  только через интерфейсы (ports), реализуемые в infrastructure.
- Принятые паттерны и анти-паттерны — [../architecture/patterns.md](../architecture/patterns.md).
- Отступления от границ слоёв допускаются только через новый ADR.

## Последствия

**Положительные:**
- Предсказуемая структура: и человек, и агент по типу задачи однозначно
  определяют, в каком слое писать код.
- Домен тестируется юнит-тестами без БД и сети
  (см. [../architecture/testing-strategy.md](../architecture/testing-strategy.md)).
- Замена инфраструктуры (БД, брокер, внешний API) локализована в одном слое.
- Единые правила во всех проектах группы — дешёвое переключение между ними.

**Отрицательные:**
- Больше кода-обвязки: интерфейсы, DTO, мапперы между слоями; простые CRUD-фичи
  требуют больше файлов.
- Порог входа для разработчиков без опыта CA.

**Риски:**
- «CA на бумаге»: границы декларированы, но не соблюдаются. Митигация: проверка
  направления зависимостей на ревью и линтером — TODO(template): укажите

  ArchUnit, dependency-cruiser).
- Переусложнение мелких сервисов. Митигация: для утилит вне ядра допускается
  упрощённая структура — фиксируется отдельным ADR.
