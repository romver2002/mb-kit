# Схема frontmatter

Каждый `.md` в `memory-bank/` начинается с YAML-frontmatter. Это и есть
версионирование «в тексте»: агент не видит git log, поэтому версия, дата и статус
живут в самом документе. Валидатор проверяет схему автоматически.

## Обязательные поля

```yaml
---
status: active            # draft | active | archived
version: "1.3"            # мажор.минор; поднимать при содержательных правках
updated: 2026-07-05       # дата последней содержательной правки (ISO, YYYY-MM-DD)
verified_commit: ""       # хеш коммита кода, на который документ проверен (может быть пустым)
owner: ""                 # ответственная роль/человек (может быть пустым)
---
```

### status
- `draft` — черновик: наполняется, на него нельзя опираться без проверки.
- `active` — действующий документ, источник истины.
- `archived` — устарел, хранится для истории; агент не использует его как истину.

### version
- Строка `"мажор.минор"` в кавычках.
- **Минор +1** — содержательная правка (изменился смысл, добавлен/удалён факт).
- **Мажор +1, минор в 0** — документ существенно переработан или изменил охват.
- Опечатки, форматирование, правка ссылок — версию **не** поднимают
  (и `updated` не трогают): история таких правок — в git.
- Версию и дату проставляет автоматика: `scripts/bump_frontmatter.py`
  в git pre-commit (включение однократно на каждом клоне:
  `git config core.hooksPath scripts/git-hooks`) и рубеж в CI, который не пропустит
  правку тела без bump. Вручную поднимать не нужно; ручной bump (например, мажорный
  при переработке) автоматика уважает и не дублирует.
- Исключение — `current/`: их frontmatter ведёт `scripts/mb_log.py` (он сам
  поднимает version/updated при записи), поэтому CI-рубеж их не форсит.
  Если pre-commit не включён, недостающий bump поймает CI на PR.

### updated
- Дата последней содержательной правки. Меняется вместе с `version`.

### verified_commit
- Короткий хеш коммита кода, на момент которого документ проверен на соответствие
  реальности. Позволяет агенту понять отставание: `git log <hash>..HEAD -- <каталог>`
  покажет, что менялось в коде после последней сверки.
- Заполняется при сверке документа с кодом (см. [lifecycle.md](lifecycle.md)).
  Пустая строка = сверка не проводилась.

## Опциональные поля

```yaml
derived_from:             # канонические документы, из которых выведен этот
  - ../domain/domain-model.md
supersedes: "0003"        # только для ADR: номер решения, которое заменяет
decision_status: accepted # только для ADR: proposed | accepted | superseded | rejected
tags: [billing, api]
```

- `derived_from` — если факт в этом документе производен от другого, правится
  сначала канонический документ (правило «сверху вниз», см. [lifecycle.md](lifecycle.md)).
- `decision_status` — жизненный цикл ADR; сам файл ADR после `accepted`
  по существу не редактируется (см. [../decisions/README.md](../decisions/README.md)).

## Чего в frontmatter НЕ делаем

- **Не ведём changelog внутри документа.** История правок — это git.
  Frontmatter отвечает только на вопросы «насколько свежо» и «можно ли верить».
- Не дублируем содержимое (заголовок, описание) — это тело документа.
