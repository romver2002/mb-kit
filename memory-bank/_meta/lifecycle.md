# Жизненный цикл документов

Правила, которые не дают базе превратиться в мусор. Обязательны для людей и агентов.

## Принципы

1. **SSoT — один факт живёт в одном документе.** Остальные документы ссылаются
   на него, а не копируют. Кто каким фактом владеет — [governance.md](governance.md).
2. **Сверху вниз.** Если документ помечен `derived_from`, правка начинается
   с канонического источника, затем синхронизируются производные.
3. **Ссылка дешевле копии.** Сомневаешься, копировать или сослаться — ссылайся.
4. **База меняется через PR** и ревьюится как код. Исключение — `current/`:
   агент обновляет его в рабочих ветках по ходу задач.

## Обновление документа

При содержательной правке (изменился смысл, добавлен/удалён факт):

1. `version`/`updated` поднимет автоматика (pre-commit/CI,
   `scripts/bump_frontmatter.py`) — вручную не трогать
   (правила — в [frontmatter.md](frontmatter.md)).
2. Проверить производные документы (поиск по `derived_from` и обратным ссылкам) —
   синхронизировать в том же PR.
3. Добавил/удалил/переименовал файл → обновить индекс
   [../README.md](../README.md) в том же PR.
4. Прогнать `python scripts/validate_memory_bank.py`.

Опечатки/форматирование — правятся без изменения `version`/`updated`.

## Сверка с кодом (verified_commit)

Сверка обязательна для `architecture/`, `domain/` и `ops/`; для `product/`
не требуется (с кодом не сверяется — только квартальный просмотр владельцем).

Раз в спринт (или после крупного мерджа) владелец раздела — сам или силами агента:

1. Смотрит, что менялось в коде с последней сверки:
   `git log <verified_commit>..HEAD --oneline -- <каталоги раздела>`.
2. Сверяет документ с текущим кодом, правит расхождения.
3. Проставляет свежий `verified_commit` (даже если правок не было — сверка была).

Промпт для агента:

```text
Сверь memory-bank/architecture/overview.md с текущим кодом. В frontmatter указан
verified_commit — изучи изменения кода после него (git log/diff по каталогам слоёв)
и предложи правки документа. Противоречия перечисли отдельно, не исправляй молча.
```

## Противоречия

Противоречие между документами или документом и кодом — **дефект базы**.

- Человек: исправляет в каноническом источнике немедленно (PR).
- Агент: если исправление НЕ входит в текущую задачу — не чинит молча, а фиксирует
  в [../current/progress.md](../current/progress.md) (раздел «Расхождения»)
  и сообщает пользователю. Если правка документа — часть задачи, чинит по правилу
  «сверху вниз».

## Архивирование

- Документ потерял актуальность → `status: archived` + строка в начале тела:
  `> Архив: заменён <ссылка>` (или причина). Из индекса ссылка остаётся
  с пометкой «архив» либо убирается, если не нужна для истории.
- ADR не архивируются — у них свой цикл `decision_status`
  (см. [../decisions/README.md](../decisions/README.md)).
- Файлы `current/` не архивируются: устоявшееся из них переносится в постоянные
  разделы, отработанное — просто удаляется из файла (история остаётся в git).

## Регулярное обслуживание (раз в спринт)

Триггер — еженедельный отчёт CI (scheduled-прогон валидатора создаёт issue
с предупреждениями на владельца из [governance.md](governance.md)); чек-лист
ниже — содержание этого разбора, а не отдельный ритуал по памяти.

- [ ] Просмотреть `current/`: перенести устоявшееся, вычистить отработанное.
- [ ] Пройтись по предупреждениям валидатора об устаревании.
- [ ] Сверить 1–2 стабильных документа с кодом (ротацией), обновить `verified_commit`.
- [ ] Просмотреть автопамять агентов (`/memory` в Claude Code) — полезное перенести в базу.
- [ ] Взглянуть на `python scripts/kb_usage_report.py` — какие файлы базы мертвы.

## Критерии сворачивания

Записаны заранее, чтобы решение «признать провал» не было политически дорогим:

- Три отчёта CI подряд проигнорированы (issue без реакции) → базу не читают люди;
  сократить объём до ядра (current/ + glossary + local-dev) и пересобрать ритуалы.
- `kb_usage_report.py` показывает < 30% сессий с чтением базы месяц подряд →
  агенты работают мимо базы; чинить протокол (AGENTS.md, hooks), а не наполнять базу.
- Раздел мёртв (0 чтений и 0 правок за квартал) → слить в родительский README
  или удалить. Пустая заготовка хуже отсутствия файла.
