# Процесс: рефакторинг

Пошаговый процесс для агента. Протокол сессии — в [../../AGENTS.md](../../AGENTS.md).
Рефакторинг не меняет наблюдаемое поведение; если поведение меняется —
это фича ([new-feature.md](new-feature.md)) или баг ([bug-fix.md](bug-fix.md)).

## 0. Перед началом прочитай

- [ ] [../current/active-context.md](../current/active-context.md) — не пересекается ли рефакторинг с текущими работами (риск конфликтов).
- [ ] [../architecture/patterns.md](../architecture/patterns.md) — целевые паттерны и анти-паттерны.
- [ ] [../architecture/overview.md](../architecture/overview.md) — границы слоёв, которые нельзя нарушить.
- [ ] [../architecture/testing-strategy.md](../architecture/testing-strategy.md) — чем будет доказана эквивалентность поведения.
- [ ] [../decisions/README.md](../decisions/README.md) — нет ли ADR, который уже фиксирует текущее устройство (тогда менять его можно только новым ADR).

## 1. Зафиксируй цель и границы

Запиши до первого изменения кода (в задаче/PR и в
[../current/active-context.md](../current/active-context.md)):

1. **Цель** — какую конкретную проблему устраняем (дублирование, нарушение слоя,
   god-object, связность). «Сделать красиво» — не цель.
2. **Границы** — какие модули/каталоги трогаем.
3. **Что НЕ трогаем** — явный список: публичные контракты API, схема БД,
   поведение, соседние модули «за компанию». TODO(template): постоянные
   запретные зоны проекта (например, каталог legacy/).
4. Критерий остановки: цель достигнута → стоп, даже если видно «ещё улучшения».
   Новые идеи — отдельной записью в [../current/progress.md](../current/progress.md).

## 2. Тесты до

1. Прогони весь набор — базовая линия должна быть зелёной. Красные тесты —
   сначала разберись с ними (это может быть чужой баг → [bug-fix.md](bug-fix.md)).
2. Затронутый код слабо покрыт → добавь characterization-тесты, фиксирующие
   текущее поведение, до любых изменений.

## 3. Шаги малыми порциями

1. Один шаг = одно механическое преобразование (переименование, extract,
   перемещение, инверсия зависимости).
2. После каждого шага: сборка успешна, тесты зелёные. Красное после шага —
   откати шаг, а не чини «по пути».
3. Коммиты — по шагам, по [../architecture/git-workflow.md](../architecture/git-workflow.md);
   смешивать рефакторинг с функциональными изменениями в одном коммите нельзя.

## 4. Сверка с архитектурой

- [ ] Направление зависимостей после рефакторинга — только внутрь
  (проверь по [../architecture/overview.md](../architecture/overview.md)).
- [ ] Не появился анти-паттерн из [../architecture/patterns.md](../architecture/patterns.md).
- [ ] Меняются границы слоёв, вводится/выводится паттерн → **черновик ADR**
  по [../decisions/0000-template.md](../decisions/0000-template.md)
  (`decision_status: proposed`); статус accepted ставит человек —
  см. [../_meta/governance.md](../_meta/governance.md). До согласования
  ограничься минимальным объёмом изменений.

## 5. Обнови базу и current/

- [ ] Рефакторинг изменил то, что описано в architecture/ (структура слоёв,
  паттерн, стек) → обнови соответствующий документ в том же PR.
- [ ] [../current/active-context.md](../current/active-context.md) — итог: что сделано, что осталось за границами.
- [ ] [../current/progress.md](../current/progress.md) — отложенные идеи улучшений, найденные расхождения.

## Definition of Done

- [ ] Цель и границы были зафиксированы до начала; границы не нарушены.
- [ ] Наблюдаемое поведение не изменилось: весь набор тестов зелёный, новых функциональных изменений нет.
- [ ] Сверка с patterns.md и overview.md пройдена; при смене границ/паттернов — черновик ADR создан и озвучен.
- [ ] Затронутые architecture-доки и `current/` обновлены; `python scripts/validate_memory_bank.py` проходит.

## Пример (замените своим)

Цель: use case `PlaceOrder` напрямую вызывает SMTP-клиент — нарушение слоя.
Границы: модуль ordering/application; не трогаем шаблоны писем и публичный API.
Тесты до: добавлен characterization-тест «заказ размещён → письмо отправлено».
Шаги: (1) извлечь интерфейс `NotificationPort` в application; (2) реализацию
`SmtpNotificationAdapter` перенести в инфраструктуру; (3) перевязать DI.
Границы слоёв изменились → черновик ADR «Уведомления через порт» предложен
человеку. architecture/patterns.md дополнён примером порта, current/ обновлён.
