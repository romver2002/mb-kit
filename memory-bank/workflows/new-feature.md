# Процесс: новая фича

Пошаговый процесс для агента. Протокол сессии (что читать всегда, как обновлять
базу) — в [../../AGENTS.md](../../AGENTS.md), здесь только специфика фичи.

## 0. Перед началом прочитай

- [ ] [../current/active-context.md](../current/active-context.md) — не конфликтует ли фича с текущим фокусом.
- [ ] [../domain/domain-model.md](../domain/domain-model.md) и [../domain/business-rules.md](../domain/business-rules.md) — какие сущности и правила задевает фича.
- [ ] [../domain/glossary.md](../domain/glossary.md) — термины из формулировки задачи; незнакомый термин = вопрос человеку.
- [ ] [../architecture/overview.md](../architecture/overview.md) и [../architecture/patterns.md](../architecture/patterns.md) — куда фича ложится по слоям.
- [ ] [../product/roadmap.md](../product/roadmap.md) — если непонятен приоритет или контекст «зачем».

## 1. Уточни требования

Не начинай кодить, пока нет ответов (спроси у человека, чего не хватает):

1. Кто пользователь фичи и какой сценарий она закрывает?
2. Какие граничные случаи и что считается ошибкой (валидация, конфликты, лимиты)?
3. Есть ли новые бизнес-правила? Их источник (человек/документ)?
4. Что явно НЕ входит в объём (out of scope)?
5. Как поймём, что фича работает (наблюдаемый критерий приёмки)?

## 2. Крупная фича → spec-kit

Фича крупная, если верно хотя бы одно: задевает больше одного bounded context,
вводит новый агрегат, требует больше ~3 новых use case или миграции данных.

## 3. Реализация по слоям (изнутри наружу)

1. **Домен** — сущности, value objects, инварианты. Без зависимостей наружу.
2. **Use case (application)** — сценарий, порты (интерфейсы) для внешнего мира.
3. **Адаптеры** — контроллеры/презентеры, реализации портов (БД, API, очереди).
4. **Композиция** — регистрация зависимостей, конфигурация, маршруты.

На каждом шаге сверяйся с [../architecture/patterns.md](../architecture/patterns.md);
направление зависимостей — только внутрь. Новое архитектурное решение —
черновик ADR по [../decisions/0000-template.md](../decisions/0000-template.md).

## 4. Тесты

По [../architecture/testing-strategy.md](../architecture/testing-strategy.md):
домен и use case покрываются обязательно, адаптеры — по стратегии проекта.
Прогони весь набор, не только новые тесты.

## Пример (замените своим)

Фича «Промокод при оформлении заказа»: уточнили правила стекирования скидок
(вопрос 2) — промокод не суммируется с акцией. Фича мелкая (один контекст
Ordering, один use case) — без spec-kit. Домен: value object `PromoCode`,
инвариант «скидка ≤ 100%»; use case `ApplyPromoCode` + порт `PromoCodeRepository`;
адаптер — реализация на {{STACK}}. Тесты: юнит на инвариант и стекирование,
интеграционный на репозиторий. В базу: правило стекирования → business-rules.md,
термин «промокод» → glossary.md. current/ обновлён, статус — «готово к ревью».
