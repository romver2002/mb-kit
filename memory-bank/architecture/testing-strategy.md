# Стратегия тестирования

Как {{PROJECT_NAME}} проверяет корректность по слоям Clean Architecture
([overview.md](overview.md)). Когда какие тесты обязательны по процессу —
[../workflows/new-feature.md](../workflows/new-feature.md),
[../workflows/bug-fix.md](../workflows/bug-fix.md).

## Пирамида по слоям

| Уровень | Слой | Что проверяем | Изоляция |
|---|---|---|---|
| unit | domain | инварианты сущностей, доменные сервисы, бизнес-правила ([../domain/business-rules.md](../domain/business-rules.md)) | без моков инфраструктуры вообще — домен не знает о ней по построению |
| unit | application | сценарии use case: ветвления, ошибки, проверка прав | порты подменяются in-memory фейками/моками |
| integration | infrastructure | репозитории и адаптеры против реальной БД/брокера | реальная инфраструктура: testcontainers или локальный docker |
| e2e | весь стек | критические пути пользователя: TODO(template): перечислить 3–5 (напр. регистрация, оформление заказа, оплата) | окружение, близкое к prod |

Чем ниже слой — тем больше тестов; e2e — только критические пути, не «всё подряд».

## Что покрывается обязательно

- Каждое бизнес-правило из business-rules.md — unit-тестом; ссылка на правило
  в имени или докстринге теста.
- Каждый use case — happy path плюс каждый класс ошибок.
- Каждый багфикс — регрессионный тест, падающий до фикса
  ([../workflows/bug-fix.md](../workflows/bug-fix.md)).
- Каждая реализация репозитория — интеграционный тест контракта интерфейса.

## Правила

- Тест, зависящий от порядка выполнения, реального времени или сети, — дефект.
- Flaky-тест не перезапускаем «до зелёного», а чиним; если чинить сейчас нельзя —
  помечаем и заводим задачу.
- Тестовые данные — через фабрики/фикстуры: TODO(template): где лежат.
- Секреты и реальные ПДн в тестовых данных запрещены ([security.md](security.md)).
